NEXUS CITY OS
Statement of Work · Outline60-DAY SHADOW PILOT

A 60-day Decision Audit. Zero hardware, zero risk. It never turns a key.

Every decision verified, certified, and refusable.

This outline scopes a fixed-fee, 60-day shadow deployment of Nexus City OS against your city's live public feeds. The platform runs its full detection and recommendation pipeline in Shadow mode — it logs every would-be signal-timing plan and signs a safety certificate for each, but executes nothing. At the end you receive a signed Decision Audit report: counterfactual evidence of what the AI would have recommended, versus what actually happened.

Scope

In scope

  • Software-only deployment on one machine in the TOC (or a container host)
  • Adapter onboarding to your existing camera, transit, dispatch and weather feeds
  • Shadow-mode capture: would-be plans logged & certified, never executed
  • Operator console access (read + acknowledge + notes) for your staff
  • HMAC-signed safety certificate on every drafted plan
  • Decision Audit report + grant / NIST AI RMF packs at exit

Out of scope

  • No hardware install — no controllers, cabinets, sensors or cameras purchased or touched
  • No signal actuation — there is no code path to change timing in Shadow mode
  • No integration with the physical signal controller network
  • No raw video ingestion or storage; no face / plate recognition
  • No licensed third-party probe data (zero external API keys required)
  • Live-execution and NTCIP controller bridge — roadmap, not this pilot
Data handled
DataHandling
Camera frames Analyzed at the edge for incident detection; redacted before ingestion — unredacted telemetry is rejected and the rejection is audit-logged. Frames kept as detection-time evidence tied to a specific incident, released on a retention schedule. No raw video enters the platform.
Transit GPS Public GTFS-RT vehicle positions used as anonymous congestion probes. No rider data.
911 dispatch Public real-time feed for the emergency layer. Street addresses are redacted from shareable reports — aggregate counts and role labels only; full per-incident detail stays in the access-gated legal export.
Operator decisions Approve / reject labels recorded by role, keyed to plan ID — never operator email. Operator notes never appear in shareable reports.
Weather / hazard Public NWS feeds; recorded in each plan's provenance. No PII.
Timeline & deliverables
WEEK 1 Adapter onboardingMap your city's feeds onto the platform's city graph via the adapter SDK. Stand up the shadow instance; verify feeds and the hash-chained audit trail. No hardware, no downtime.
WEEKS 2–9 Shadow captureThe platform runs continuously in Shadow mode: detects incidents, drafts signal-timing plans, runs the safety gate, signs a certificate per plan, and logs everything. Your operators use the console read-only. History retention is set to the full pilot window (60/90 days) so nothing is pruned.
WEEK 9+ Decision Audit reportDelivery of the signed evidence report — plans logged / certified / refused / abstained, matched before/after congestion windows, chain-verification proof — plus the SS4A/SMART grant packet and the NIST AI RMF conformity pack.
Exit criteria
1Signed Decision Audit report delivered — counterfactual evidence with methodology stated inline ("directional signal, not ground truth"), honest about the capture start date.
2Audit chain verifies intact — every governance event hash-chained end to end; verification proof included in the report.
3Every drafted plan carries a valid, issuer-verifiable certificate — confirmable through the Verification API.
4Grant + NIST AI RMF packs delivered — print-ready HTML exhibits.
5Zero signal actuations occurred — provable from the audit trail; Shadow mode never executed.
6Go / no-go decision — the city decides whether Advisory mode earns its place. Clean teardown at any time; the city keeps all exported evidence.

Risk posture — read this

Operational risk is structurally zero. Shadow mode has no code path that touches signal hardware. The platform observes, drafts and certifies; it does not act. Human operators remain the only actuators of the physical network throughout.

Deployment is boring by design. One Python standard-library process with a local SQLite store — no message broker, no container fleet, no external database. It survives restarts with its mode, incidents and audit chain intact.

This is a first pilot. The live Seattle instance is a running reference demo, not a paying deployment. We will not represent deployed cities that do not exist.
